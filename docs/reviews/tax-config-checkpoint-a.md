# Tax config temporal table and as-of calculations - Checkpoint A design

**Status:** Tier-3 Checkpoint A approved 2026-07-15. Checkpoint A and B rows
will land with the implementation commit per L-0019.

## Goal and decisions

This unit adds a durable, day-level source for confirmed Romanian tax
parameters without inventing prior-year values. It adds:

1. A temporal `tax_config` table for scalar rates, scalar RON amounts, and one
   versioned investment-CASS bracket set.
2. A normalized child table for the four rows in that bracket set.
3. Exact, integer-only salary and dividend-tax calculation services that
   resolve every input as of the calculation's driving date.
4. A reproducible 2026 seed containing only the values in the owner's
   accountant-confirmation brief.

The date model is **half-open**: a row covers `[valid_from, valid_to)`. A null
`valid_to` means infinity. If a new rate starts on `2026-05-01`, the old row's
`valid_to` is `2026-05-01`; April 30 resolves old and May 1 resolves new. This
avoids overlapping ownership of a boundary day.

No row before 2026 is created. A request outside known coverage fails; it does
not use the nearest row, the first row, or the current row.

## 1. Proposed schema

### Enums

```text
tax_config_parameter:
  cas_employee_rate
  cass_employee_rate
  cam_employer_rate
  income_tax_rate
  dividend_tax_rate
  minimum_wage
  personal_deduction
  cass_investment_brackets

tax_config_value_kind: rate_bps | amount_minor | bracket_set
tax_config_status: confirmed | estimate
```

The parameter enum is intentionally closed. Adding a tax parameter therefore
requires a migration and an explicit TypeScript mapping from parameter to
value kind; a misspelling cannot create a second logical series.

### `tax_config`

| Column | Type | Rule |
|---|---|---|
| `id` | UUID | Primary key, generated by Postgres |
| `parameter` | `tax_config_parameter` | Required |
| `value_kind` | `tax_config_value_kind` | Required; must match the parameter's declared kind |
| `rate_bps` | integer nullable | Set only for rate parameters; 25% = 2500, 2.25% = 225 |
| `amount_minor` | bigint nullable | Set only for amount parameters; integer bani |
| `valid_from` | date | Required, inclusive |
| `valid_to` | date nullable | Exclusive; null is open-ended |
| `status` | `tax_config_status` | Required |
| `source` | text | Required, non-blank provenance note |
| `created_at`, `updated_at` | timestamptz | Existing `timestamps` helper |

There is no soft delete. Temporal rows are audit-relevant configuration;
superseding a value closes its window and inserts a successor. Correcting a
bad row is an explicit update under the same invariant checks, not a hidden
live/deleted branch.

Database checks:

- `valid_to IS NULL OR valid_to > valid_from`.
- `source <> ''` after trimming.
- Exactly one value shape: rate rows have `rate_bps` only, amount rows have
  `amount_minor` only, bracket-set rows have neither scalar.
- Rates are between 0 and 10,000 bps; amounts are non-negative safe integer
  minor units.
- Unique `(parameter, valid_from)`.
- A parameter-to-kind `CHECK` mirrors the exhaustive TypeScript map, so a rate
  cannot be stored as an amount even through direct SQL.

Use `moneyMinor` (`bigint` in number mode) for amount values. The existing
helper avoids the int4 ceiling while preserving the project invariant that
money is integer minor units. Rates remain int4 basis points because every
confirmed rate has at most two decimal percentage digits and basis points
represent all of them exactly.

### `tax_config_cass_investment_brackets`

The investment brackets are a set, not a collection of unrelated scalar
parameters. Store them as children of the temporal
`cass_investment_brackets` parent:

| Column | Type | Rule |
|---|---|---|
| `id` | UUID | Primary key |
| `tax_config_id` | UUID FK | Required, cascade delete; parent must be `bracket_set` |
| `ordinal` | integer | Required, unique per parent |
| `lower_minor` | bigint | Inclusive annual-income lower bound |
| `upper_minor` | bigint nullable | Exclusive upper bound; null only on final bracket |
| `base_minor` | bigint | CASS assessment base for the bracket |
| `cass_minor` | bigint | Fixed CASS due for the bracket |
| timestamps | timestamptz | Existing helper |

All monetary fields are integer bani. Row checks require non-negative values
and `upper_minor > lower_minor` when present. A set validator requires ordinal
zero to start at zero, every upper bound to equal the next lower bound, and
only the final upper bound to be null. The annual cumulative CASS calculation
does not consume this table in this unit; the rows are stored now so Phase 5
does not have to reverse-engineer thresholds from parameter names.

## 2. Window invariant

The invariant is scoped to the known series: before a parameter's earliest
row, coverage may be absent and the resolver fails. From the earliest row
forward, windows are contiguous, non-overlapping, and end in exactly one
open-ended row.

Use two database protections in migration SQL:

1. A deferrable GiST exclusion constraint on
   `(parameter WITH =, daterange(valid_from,
   coalesce(valid_to, 'infinity'::date), '[)') WITH &&)`. This requires
   `btree_gist` and prevents overlapping windows, including concurrent writes.
2. A `DEFERRABLE INITIALLY DEFERRED` constraint trigger that sorts the affected
   parameter's rows at transaction end and rejects any adjacent pair where
   `current.valid_to <> next.valid_from`, or a final row whose `valid_to` is
   non-null. Deferral lets a legitimate replacement close the old row and add
   the new row atomically without an invalid intermediate state.

The seed and future settings write service also call pure validators before
SQL, producing typed error codes rather than exposing constraint prose. The
database remains the last line of defense for direct SQL and concurrency.
Updating a row's parameter validates both the old and new series.

The same pure validation pass checks each bracket set's internal ranges. The
seed runs all validators before inserting anything, so an overlap or gap aborts
the seed transaction.

## 3. Confirmed 2026 seed

The seed remains on the existing `npm run db:seed` path and inside its database
transaction. A shared typed constant supplies the rows to both the seed and
tests; there is no migration-time data insert and no second provisioning path.
The isolated test runner proves migrate-then-seed from an empty test database.

All parent rows have `status = confirmed`. Their source is
`accountant letter 2026-07`, except that the personal-deduction source appends
the required applicability note.

Because the current live database is non-empty and the seed correctly refuses
to rerun there, the schema migration does **not** insert these rows into live
data. The implementation verifies them through migrate-then-seed on the
isolated test database. Populating an existing database would be a separate,
owner-approved data operation (or the later reproducible reset), never a
hidden migration side effect.

### Existing `tax_rules` boundary

The repo already has `tax_rules`, and `tax_accruals.tax_rule_id` references it.
It also carries the still-unconfirmed micro-revenue placeholder and is read by
the current salary/dividend ledger flows. Replacing it would widen this unit
into accrual-link migration, ledger flow actions, dashboard joins, and policy
for the unconfirmed micro rate.

Recommendation for this unit: **add `tax_config` alongside `tax_rules` and do
not alter the ledger write path or existing accrual foreign key.** The new
resolver/calculators use only `tax_config`. The legacy flow remains explicitly
legacy until a separately scoped Tier-3 integration decides how accruals refer
to config rows and removes the duplicate placeholder source. This avoids a
silent dependent-structure migration under L-0012, but it also means this unit
does not claim that the existing salary/dividend booking UI has switched to the
new calculator. The existing `tax_rules` placeholder seed remains unchanged in
this boundary; it is not copied into `tax_config`. Checkpoint A approval
includes approval of this boundary.

## 4. Rounding rules and fixture proof

### Evidence

ANAF's D112 validation specification states that contributions use arithmetic
rounding to whole lei: a fractional part of at least 0.5 adds one to the
integer part; a positive contribution below one leu is raised to one leu. The
2025 D112 validation structure repeats that rule and defines calculated fields
as positive integers obtained by mathematical rounding:

- `https://static.anaf.ro/static/10/Anaf/Declaratii_R/AplicatiiDec/structura_D112_0125_250325.pdf`
- `https://static.anaf.ro/static/10/Anaf/Declaratii_R/AplicatiiDec/structura_D112_0825_081025.pdf`

The current D100 page likewise publishes integer-valued validation material
for state-budget liabilities; this design applies the same whole-leu arithmetic
rule to dividend withholding rather than retaining bani that cannot be declared:

- `https://static.anaf.ro/static/10/Anaf/Declaratii_R/100.html`

The apparent opposite directions in the payslip are therefore not two rules.
They are the two sides of one nearest-whole-leu, half-up rule.

### Per-figure policy

| Figure | Rule |
|---|---|
| CAS | Apply bps to gross, arithmetic half-up to whole RON; positive sub-1-RON result becomes 1 RON |
| CASS employee | Same as CAS |
| CAM employer | Same as CAS |
| Personal deduction | Stored amount, no arithmetic rounding |
| Taxable base | `max(0, gross - CAS - CASS - deduction)` in minor units; no extra rounding |
| Salary income tax | Apply bps to taxable base, arithmetic half-up to whole RON |
| Dividend withholding tax | Apply bps to gross dividend, arithmetic half-up to whole RON |

Dividend whole-leu rounding rests on ANAF D100 material, not on the July-2026
accountant confirmation. Keep it on the open-questions list for the next
accountant contact even after the implementation fixture is green.

Implementation uses integer/`bigint` rational arithmetic, not `number`
multiplication and not `Math.round` on a binary float. For an amount in bani
and a bps rate, divide the exact product by `1_000_000` to obtain whole RON,
add half the denominator for half-up rounding, then multiply the integer RON
result by 100 to return minor units. Reject results outside JS safe-integer
range.

Payslip proof, all stored-minor-unit exact:

```text
gross       450,000 bani
CAS         roundRON(450,000 * 2500 bps) = 112,500 bani
CASS        roundRON(450,000 * 1000 bps) =  45,000 bani
deduction                                      62,800 bani
taxable base = 450,000 - 112,500 - 45,000 - 62,800
             = 229,700 bani
income tax  roundRON(229,700 * 1000 bps) =  23,000 bani
CAM         roundRON(450,000 *  225 bps) =  10,100 bani
```

Thus `229.70 RON -> 230 RON` and `101.25 RON -> 101 RON` exactly match the
accountant fixture without tax-specific up/down guesses.

## 5. Service API and driving dates

All services are locale-free and throw typed `LedgerValidationError` codes.
The display edge continues to translate `errors.*`; no raw service prose is
returned.

```ts
type TaxConfigParameter = /* enum-derived union */;

resolveTaxConfig(
  parameter: TaxConfigParameter,
  date: string,
  executor?: DbExecutor,
): Promise<ResolvedTaxConfig>;

calculateSalary(input: {
  grossMinor: number;
  payPeriodDate: string;
  isBaseFunction: boolean;
  dependents: number;
}): Promise<SalaryTaxResult>;

calculateDividendTax(input: {
  grossDividendMinor: number;
  distributionDate: string;
}): Promise<DividendTaxResult>;
```

`resolveTaxConfig` validates canonical `YYYY-MM-DD`, selects
`valid_from <= date AND (valid_to IS NULL OR date < valid_to)`, and requires
exactly one row. No row throws the distinct code `tax.configCoverageMissing`
with `{ parameter, date }`. More than one row is treated as invariant
corruption and throws `tax.configWindowInvalid` even though the exclusion
constraint should make it unreachable.

`calculateSalary` resolves CAS, CASS, CAM, income tax, and personal deduction
using the **pay-period date**. A code comment will state that this date is the
salary calculation's legal driving date; callers must pass the date belonging
to the pay-period month, not the booking timestamp. The existing UI's month-end
date is a valid caller representation, but wiring that UI is outside this unit.
All five resolutions run through one caller-owned database transaction at
repeatable-read isolation, so a concurrent config replacement cannot produce a
mixture of versions inside one result.

Because only one personal-deduction case is confirmed, the calculator refuses
to apply 628 RON unless gross is exactly 4500 RON, `isBaseFunction` is true,
and `dependents` is zero. Other cases throw
`tax.personalDeductionContextUnsupported`; they do not receive zero, 628, or
an interpolated amount. This guard implements the source-note limitation, not
a deduction formula or grid.

For dates before `2026-05-01`, deduction resolution runs before applicability
validation and therefore throws `tax.configCoverageMissing`, not
`tax.personalDeductionContextUnsupported`. That ordering is intentional: no
deduction row exists for the date, so the temporal-data gap is the first and
more precise failure. The unsupported-context code applies only when a
deduction row covers the date but the gross/base-function/dependent facts are
outside its confirmed scope.

`SalaryTaxResult` returns gross, CAS, CASS, CAM, deduction, taxable base, and
income tax in minor units, plus the applied config row IDs/statuses so a future
edge can mark any estimate honestly. The aggregate status is `estimate` if any
input row is estimated, otherwise `confirmed`.

`calculateDividendTax` resolves `dividend_tax_rate` using the
**distribution date**, documented in code as its legal driving date. It returns
gross, tax, net, and the applied config metadata. It does not calculate annual
investment CASS.

New catalog-complete codes proposed:

- `tax.configCoverageMissing`
- `tax.configWindowInvalid`
- `tax.configValueInvalid`
- `tax.personalDeductionContextUnsupported`

They are added to `ERROR_CODES` and both `errors.tax` catalogs. The existing
compile-time `ERROR_CODES`/English-catalog equality guard remains the
completeness proof, verified with cache-cleared tsc under L-0013.

## 6. Isolated test plan

Add a dedicated runner following the reversal suite's established protection:

- Require both `DATABASE_URL` and `TEST_DATABASE_URL`.
- Refuse if their normalized identities match.
- Require the test database name to end in `_test`.
- Reset only that test database, migrate it, run the real seed path, then run
  tax-config tests. No test writes to the live dev database.

Required tests:

1. **Seed shape:** exactly eight confirmed parent rows and four bracket child
   rows; exact source notes; no row begins before 2026; each parameter series
   passes continuity validation.
2. **Payslip fixture:** gross 450,000 bani at a May-2026 pay-period date, base
   function, zero dependents -> CAS 112,500; CASS 45,000; CAM 10,100;
   deduction 62,800; taxable base 229,700; income tax 23,000.
3. **Rounding directions:** explicit assertions preserve CAM `101.25 -> 101`
   and income tax `229.70 -> 230`; a half-leu fixture proves `.50` rounds up.
4. **Dividend:** a 2026 distribution resolves 1600 bps and returns exactly
   16% after whole-RON rounding; a fractional fixture locks the rounding rule.
5. **Missing coverage:** every resolver call before its earliest row throws
   `tax.configCoverageMissing` with exact `{ parameter, date }`; no fallback.
6. **Overlap:** a synthetic two-row series with intersecting half-open ranges
   is rejected by the validator and by the deferred database constraint.
7. **Gap:** a synthetic two-row series whose first `valid_to` differs from the
   second `valid_from` is rejected by the validator and deferred trigger.
8. **Random-month boundary:** inside one test transaction, close a synthetic
   estimate rate at `2027-04-01` and insert its successor there. March 31
   resolves old; April 1 resolves new. The synthetic rows are test-only and
   disappear when the isolated database is reset.
9. **Bracket shape:** the four seeded ranges are contiguous, cover zero through
   infinity, and preserve the exact bases/CASS amounts from the brief.
10. **Personal-deduction safety:** non-4500 gross, non-base function, and any
    dependents each fail with the unsupported-context code; no formula is
    inferred.

Checkpoint B will include cache-cleared `tsc`, changed-file eslint, G1-G4,
catalog parity, migration output, isolated migrate-and-seed output, the full
tax suite, the existing service battery, and `next build`. The seed report will
state exact parent/child counts and date coverage rather than only saying green.

## 8. Scope and Tier-3 invariants

In scope: Drizzle enums/schema, one migration, seed constants/path,
locale-free tax config resolver/calculators, typed error catalogs, isolated
test runner, tests, this design doc, and A/B review-log rows at implementation
commit time.

Out of scope: settings UI, personal-deduction formula/grid, annual cumulative
CASS, import gross/withholding schema, 2024/2025 data, foreign-income FX
reporting, and migration of legacy `tax_rules`/`tax_accruals`/ledger flows.

Money remains integer minor units. This unit does not alter ledger posting
math, zero-sum enforcement, RON mirrors, tax-accrual links, soft-delete,
transaction writes, or the seed's fixed entity/account UUIDs. No booking write
bypasses `createTransaction`; this unit itself books nothing.

Checkpoint A was approved 2026-07-15. Implementation still stops at Checkpoint
B for owner review before any commit; ambiguous continuation does not waive
that STOP gate (L-0017).
